# 2. АНАЛИЗ ИСПОЛЬЗУЕМЫХ ТЕХНОЛОГИЙ И ВЫБОР СТЕКА РАЗРАБОТКИ

Выбор технологического стека определяется требованиями, сформулированными в подразделе 1.3: кроссплатформенность в связке Android и Windows, полнофункциональная работа в автономном режиме, выполнение прогнозирующей модели машинного обучения непосредственно на устройстве и облачная синхронизация данных. В настоящем разделе последовательно обоснован выбор клиентской платформы, средства локального хранения данных, серверной платформы с облачной базой данных и технологий машинного обучения. По каждому направлению рассмотрены альтернативы и приведены критерии, на основании которых принято решение.

## 2.1 Выбор клиентской платформы

Клиентское приложение должно собираться для двух целевых платформ — Android и Windows — из единого исходного кода, сохраняя производительность, близкую к нативной, и внешний вид интерфейса. Дополнительными требованиями являются единство языка программирования с серверной частью и прямая интеграция со средой выполнения моделей машинного обучения. С учётом этих условий рассмотрены три наиболее распространённых каркаса кроссплатформенной разработки: .NET MAUI, Flutter и React Native. Их сопоставление приведено в таблице 2.1.

**Таблица 2.1 — Сравнение каркасов кроссплатформенной разработки**

| Критерий | .NET MAUI | Flutter | React Native |
|---|---|---|---|
| Язык программирования | C# | Dart | JavaScript / TypeScript |
| Поддержка Android | Да | Да | Да |
| Поддержка Windows (настольная) | Да (нативно, WinUI 3) | Да (рендеринг Skia) | Да (React Native for Windows) |
| Единый язык с серверной частью | Да (ASP.NET Core, C#) | Нет | Частично (Node.js) |
| Интеграция со средой ONNX Runtime | Прямая (нативная привязка для .NET) | Через слой межъязыкового взаимодействия | Через нативные модули |
| Модель сборки | Нативное приложение под каждую платформу | Нативное приложение (свой рендеринг) | Мост между средой JavaScript и нативным кодом |

На целевые платформы способны компилироваться все три каркаса, поэтому решающими стали два фактора. Во-первых, язык C# является общим для клиента и серверной части на ASP.NET Core, что позволяет совместно использовать классы передачи данных (DTO) и снижает риск рассогласования схем между клиентом и сервером. Во-вторых, модуль прогнозирования выполняется непосредственно на устройстве через среду ONNX Runtime, для которой платформа .NET предоставляет официальную нативную привязку; в случае Flutter или React Native обращение к ней потребовало бы дополнительного слоя межъязыкового взаимодействия. С учётом центральной роли локального выполнения модели выбрана платформа **.NET MAUI 9** [19], позволяющая из единого исходного кода на языке C# собирать нативные приложения для Android и Windows.

Для организации клиентского кода применён архитектурный паттерн «модель — представление — модель представления» (MVVM), реализуемый средствами библиотеки **CommunityToolkit.Mvvm 8.4** [20]: она генерирует шаблонный код на этапе компиляции по атрибутам `[ObservableProperty]` и `[RelayCommand]`, избавляя разработчика от написания однотипных конструкций. Подробно реализация паттерна рассмотрена в разделе 4.

## 2.2 Локальное хранение данных

В соответствии с принципом приоритета локальных данных все операции чтения и записи выполняются с базой данных на устройстве, а синхронизация с сервером запускается лишь по явному действию пользователя. Локальное хранилище должно работать без выделенного сервера, поддерживать транзакции и индексы, одинаково функционировать на Android и Windows и сохранять полную работоспособность в автономном режиме. Рассмотрены три встраиваемых решения: SQLite с доступом через sqlite-net-pcl, объектная база данных Realm и документоориентированная LiteDB. Их сопоставление приведено в таблице 2.2.

**Таблица 2.2 — Сравнение встраиваемых средств хранения данных**

| Критерий | SQLite (sqlite-net-pcl) | Realm | LiteDB |
|---|---|---|---|
| Модель данных | Реляционная | Объектная | Документоориентированная |
| Язык запросов | SQL | LINQ / собственный API | LINQ / собственный API |
| Кроссплатформенность Android + Windows | Да | Да | Да |
| Выделенный сервер | Не требуется | Не требуется | Не требуется |
| Лицензия | Общественное достояние | Apache 2.0 | MIT |
| Зрелость и распространённость | Очень высокая (стандарт де-факто) | Высокая | Средняя |

Все три варианта работают локально и поддерживают обе целевые платформы. Выбор сделан в пользу **SQLite** с доступом через библиотеку **sqlite-net-pcl 1.9** [21] по совокупности причин. Эта СУБД является стандартом де-факто для встраиваемых баз данных и обладает наибольшей зрелостью и проверенностью. Поддержка языка SQL позволяет выполнять нетривиальные запросы с соединением таблиц, необходимые модулю статистики и прогресса, что затруднительно в объектной и документоориентированной моделях. Наконец, СУБД распространяется свободно и не накладывает лицензионных ограничений. Подробное проектирование схемы базы данных приведено в разделе 3.

## 2.3 Серверная платформа и облачная база данных

Серверная часть обеспечивает аутентификацию пользователей и облачную синхронизацию данных между устройствами. Она реализована на программной платформе **ASP.NET Core 9** — высокопроизводительном кроссплатформенном каркасе для веб-приложений. В качестве альтернатив рассматривались каркасы Node.js (Express) и Java (Spring Boot), однако решающим стало единство языка программирования (C#) с клиентом: оно позволяет совместно использовать классы передачи данных, упрощает сопровождение и снижает риск рассогласования схем между клиентом и сервером.

Базой данных сервера служит **PostgreSQL**, доступ к которой выполняется через библиотеку **Npgsql**. Среди реляционных СУБД она выбрана за надёжность, развитую поддержку со стороны облачных платформ размещения и наличие конструкции `INSERT ... ON CONFLICT DO UPDATE WHERE`, на которой строится разрешение конфликтов синхронизации по стратегии «побеждает последняя запись» (LWW, Last-Write-Wins) [22]. Проектирование серверной части, схема данных и протокол синхронизации подробно рассмотрены в разделах 3 и 4.

## 2.4 Технологии машинного обучения

Центральный отличительный модуль системы — прогнозирование одноповторного максимума на горизонте 28 дней — представляет собой задачу регрессии на временно́м ряде тренировочных показателей. В литературе и инженерной практике сложилось несколько подходов к её решению; их сопоставление приведено в таблице 2.3.

**Таблица 2.3 — Сравнение методов прогнозирования силовых показателей**

| Метод | Прогноз будущего значения | Учёт затухания прироста | Потребность в данных | Выполнение на устройстве |
|---|---|---|---|---|
| Формульные (Эпли, Бжицкий, Лэндер) | Нет (оценка текущего 1ПМ) | — | Минимальная | Тривиальное |
| Линейная экстраполяция | Да, только линейный тренд | Нет | Низкая | Простое |
| Градиентный бустинг (LightGBM, XGBoost) | Да | Да | Средняя | Да (через ONNX) |
| Нейросети (LSTM, трансформеры) | Да | Да | Высокая | Затруднено |

Формульные методы, рассмотренные в подразделе 1.1 (формулы Эпли, Бжицкого и Лэндера), оценивают *текущий* 1ПМ по результату одного подхода, однако не позволяют предсказать его будущее значение, поскольку не учитывают динамику тренировочного процесса. Линейная экстраполяция продлевает аппроксимирующую прямую, построенную по значениям 1ПМ за несколько последних тренировок, на будущий момент времени; метод устойчив на монотонно возрастающих сериях, характерных для начинающих атлетов, и применяется в ряде приложений для построения линии тренда [15], но не способен отразить замедление прироста по мере приближения атлета к индивидуальному пределу.

Методы градиентного бустинга над деревьями решений обеспечивают высокую точность в задачах регрессии на структурированных (табличных) данных с большим числом признаков [3]. Реализации LightGBM и XGBoost [17] устойчивы к разнородным по природе признакам и пропускам в данных, а их ключевым достоинством применительно к рассматриваемой задаче является способность улавливать нелинейные зависимости между стажем атлета, уровнем нагрузки и темпом прироста силы. Нейросетевые методы (рекуррентные сети LSTM, трансформеры) способны моделировать долговременные зависимости в тренировочных сериях, но требуют значительно большего объёма размеченных данных и вычислительных ресурсов для обучения [18], что затрудняет их применение в условиях ограниченного набора данных.

С учётом изложенного выбран **градиентный бустинг в реализации LightGBM** [3] как метод, обеспечивающий наилучшее соотношение точности прогноза, скорости обучения и возможности преобразования модели в переносимый формат для выполнения непосредственно на мобильном устройстве. Модель обучается на языке Python 3.11 средствами библиотеки LightGBM, преобразуется в формат **ONNX** (Open Neural Network Exchange) и исполняется в приложении посредством среды **Microsoft.ML.OnnxRuntime 1.20.1** — кроссплатформенной библиотеки с официальной привязкой для языка C#, работающей на Android и Windows без промежуточных слоёв. Формат ONNX обеспечивает переносимость модели между средой обучения (Python) и средой выполнения (.NET). Формирование набора данных, обучение модели и её интеграция подробно рассмотрены в разделе 5.
