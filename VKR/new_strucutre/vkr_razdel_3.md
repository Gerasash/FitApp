# 3. ПРОЕКТИРОВАНИЕ ИНФОРМАЦИОННОЙ СИСТЕМЫ И БАЗЫ ДАННЫХ

## 3.1 Общая архитектура системы

Информационная система FitApp построена по трёхзвенной архитектуре и включает клиентское приложение, серверную часть и облачную базу данных. Общая структура системы и взаимодействие её компонентов приведены на рисунке 3.1.

*[Рисунок 3.1 — Общая архитектура системы FitApp — файл: `diagrams/diagram_2_1_architecture.drawio`]*

Клиентское приложение на платформе .NET MAUI устанавливается на устройства пользователя (Android и Windows) и хранит все данные в локальной базе SQLite, обеспечивая полную работоспособность в автономном режиме. Прогнозирование одноповторного максимума выполняется непосредственно на устройстве средствами среды ONNX Runtime, без обращения к сети. Серверная часть на ASP.NET Core предоставляет REST API для аутентификации и синхронизации, а постоянное хранение данных в облаке обеспечивает база PostgreSQL. Клиент и сервер обмениваются данными по протоколу HTTPS в формате JSON; синхронизация запускается по явному действию пользователя и не является обязательной для работы приложения.

## 3.2 Архитектура клиентского приложения

Клиентское приложение построено по архитектурному паттерну **«модель — представление — модель представления» (MVVM)**. Паттерн разграничивает ответственность между слоями: модель отвечает за данные и бизнес-логику, представление — за отображение, а модель представления связывает их посредством механизма привязки данных [23].

Структура проекта отражена в организации каталогов решения:

- `Models/` — сущности SQLite: `Workout`, `WorkoutExercise`, `ExerciseSet`, `Exercise`, `MuscleGroup`, `UserProfile`;
- `ViewModels/` — частичные классы, наследующие `ObservableObject`; команды и наблюдаемые свойства генерируются на этапе компиляции через атрибуты CommunityToolkit;
- `Views/` — XAML-страницы и файлы вспомогательного кода;
- `Services/` — сервисы бизнес-логики: предсказания ONNX, планировщик, синхронизация, клиент аутентификации;
- `Data/` — единственный класс доступа к базе данных `WorkoutDataBase`.

Навигация управляется встроенным в .NET MAUI механизмом маршрутизации **Shell**: переходы между страницами задаются URI-адресами через вызов `Shell.Current.GoToAsync`.

Приложение придерживается принципа приоритета локальных данных. Все операции чтения и записи выполняются с локальной базой SQLite, а синхронизация с сервером запускается исключительно по явному действию пользователя — нажатию кнопки «Синхронизировать». При отсутствии сетевого подключения ни одна функция приложения не блокируется.

Внедрение зависимостей настраивается в точке входа `MauiProgram.cs`, где регистрируются все сервисы и страницы. Класс `WorkoutDataBase` зарегистрирован как одиночка (singleton) — единственное соединение на весь жизненный цикл приложения, тогда как остальные компоненты создаются заново при каждом запросе.

## 3.3 Проектирование базы данных

Локальная база данных содержит семь основных таблиц. Схема отражает физическую модель данных, применяемую в `WorkoutDataBase.cs` (листинг схемы приведён в Приложении А).

*[Рисунок 3.2 — ER-диаграмма базы данных клиента FitApp — файл: `diagrams/diagram_2_2_er.drawio`]*

При проектировании схемы принят ряд решений, обеспечивающих корректную работу последующей синхронизации.

Удаление записей выполняется не физически, а установкой логического флага `IsDeleted` (мягкое удаление). Благодаря этому факт удаления распространяется на остальные устройства наравне с прочими изменениями: удалённая запись с более поздней меткой `UpdatedAt` замещает существующую запись по стратегии «побеждает последняя запись».

Каждая синхронизируемая сущность (`Workout`, `WorkoutExercise`, `ExerciseSet`) снабжена полем `SyncId` — строковым идентификатором в формате GUID, который генерируется клиентом при создании записи. Это поле служит первичным ключом на сервере и обеспечивает однозначное сопоставление записей между устройствами, исключая коллизии автоинкрементных идентификаторов.

Поле `UpdatedAt` (тип `DateTime`) проставляется клиентом при каждом изменении записи значением `DateTime.UtcNow`. Оно используется как условие выборки при формировании пакета исходящих изменений во время синхронизации.

Связь между тренировкой и группами мышц (отношение «многие-ко-многим») реализована через таблицу-связку `WorkoutMuscleGroups`. При обновлении тренировки прежние связи удаляются и записываются заново.

На стороне сервера поддерживаются четыре таблицы — `Users`, `Workouts`, `WorkoutExercises` и `ExerciseSets`; первичным ключом таблиц тренировочных данных служит `SyncId`. Схема сервера приведена в Приложении А.

## 3.4 Проектирование серверной части

Сервер реализован в виде минимального REST API на ASP.NET Core 9 и предоставляет три конечные точки.

**Таблица 3.1 — Эндпоинты API сервера FitApp**

| Метод | URL | Аутентификация | Описание |
|---|---|---|---|
| POST | `/auth/register` | Нет | Регистрация нового пользователя по логину и паролю; возвращает JWT-токен (30 дней) |
| POST | `/auth/login` | Нет | Аутентификация пользователя; возвращает JWT-токен (30 дней) |
| POST | `/sync` | Bearer JWT | Двунаправленная синхронизация: принимает пакет исходящих изменений клиента, возвращает пакет входящих изменений сервера |

Аутентификация реализована с помощью токенов JWT (JSON Web Token, RFC 7519 [24]) со схемой подписи HS256. Токен содержит идентификатор пользователя и срок действия (30 дней). Пароли хранятся в виде хешей, вычисляемых алгоритмом BCrypt с ценой вычисления (work factor), равной 10 [25].

Конечная точка синхронизации `POST /sync` принимает тело запроса с тремя массивами изменённых сущностей (`Workouts`, `WorkoutExercises`, `ExerciseSets`) и меткой времени `LastSyncUtc`. В ответ сервер возвращает все сущности пользователя, изменённые после момента `LastSyncUtc`, а также собственное текущее время `ServerTimeUtc`. Подробное описание протокола синхронизации приведено в подразделе 4.7.
