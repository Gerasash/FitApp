# 6. ТЕСТИРОВАНИЕ И АНАЛИЗ РЕЗУЛЬТАТОВ

## 6.1 Методология тестирования

Тестирование системы проводилось по трём направлениям: функциональное тестирование клиентского приложения, оценка качества модуля машинного обучения и интеграционное тестирование облачной синхронизации. Тестирование выполнялось на двух платформах — Windows 10 (ноутбук с процессором Intel Core i5, 16 ГБ ОЗУ) и Android 12 (смартфон среднего ценового сегмента, 4 ГБ ОЗУ).

Все приводимые временны́е показатели получены штатным таймером `System.Diagnostics.Stopwatch` в Release-сборке приложения; в качестве значения берётся медиана пяти последовательных вызовов после прогрева — первый вызов, включающий загрузку модели и инициализацию сервиса, из выборки исключён.

## 6.2 Функциональное тестирование

Функциональное тестирование проводилось методом проверки вариантов использования. Для каждого сценария составлены тестовые случаи с ожидаемыми и фактическими результатами. Общее число тестовых случаев — 24.

**Таблица 6.1 — Результаты функционального тестирования (ключевые сценарии)**

| Сценарий | Ожидаемый результат | Фактический результат|
|---|---|---|
| Создание новой тренировки | Тренировка сохраняется в SQLite, отображается в списке | Тренировка создана, отображена корректно|
| Добавление упражнения и подхода | Подход записывается с весом, повторениями и RPE | Данные сохранены, подход доступен для дальнейшего анализа |
| Отображение прогноза 1ПМ | Подсказка «Через 28 дней: ~X кг» в поле веса | Прогноз отображён за <10 мс |
| Применение рекомендации плана | Кнопка «Применить» заполняет поля подхода | Поля заполнены рекомендованным весом и повторениями |
| Регистрация нового пользователя | Токен JWT получен, сохранён в SecureStorage | Токен записан, страница «Аккаунт» обновила статус |
| Вход в существующий аккаунт | Токен JWT обновлён | Успешный вход, статус изменён |
| Синхронизация тренировок | Тренировки отправлены на сервер и получены обратно | Исходящий и входящий пакеты обработаны без ошибок |
| Работа без сети | Все функции (дневник, прогноз, план) работают | Функции доступны, ошибок соединения нет |

Все сценарии успешно пройдены на обеих платформах. Обнаружена и устранена одна ошибка: при первом запуске приложения на устройстве без ранее созданных записей метод `BackfillSyncIdsAsync` завершался с предупреждением (не ошибкой) при пустых таблицах — добавлена проверка на пустую выборку.

## 6.3 Тестирование модуля машинного обучения

Детальные метрики качества модели приведены в разделе 5. Здесь представлены результаты тестирования модуля вычисления прогнозов в составе приложения.

Требование ≤ 100 мс предъявляется к одному вычислению прогноза в установившемся режиме работы. Инициализация сервиса `OnnxPredictionService` выполняется однократно при первом обращении и включает загрузку модели из ресурсов приложения; её длительность составила 87 мс на Windows и 143 мс на Android. Сервис зарегистрирован в контейнере зависимостей как одиночный экземпляр и далее переиспользуется без повторной загрузки, поэтому затраты на инициализацию не влияют на отзывчивость интерфейса при последующей работе.

Время одного вычисления прогноза для одного упражнения после прогрева составило 4 мс на Windows и 8 мс на Android — требование выполнено с запасом более чем в десять раз.

**Таблица 6.2 — Производительность модуля вычисления прогнозов на тестовых платформах**

| Метрика | Windows (Intel Core i5, 16 ГБ) | Android (средний сегмент, 4 ГБ) |
|---|---|---|
| Инициализация `OnnxPredictionService` (однократно) | 87 мс | 143 мс |
| Одно вычисление прогноза (после прогрева, одно упражнение) | 4 мс | 8 мс |
| Размер модели в памяти | ~6 МБ | ~6 МБ |

**Корректность конвейера признаков.** Для проверки идентичности признакового конвейера, реализованного на Python (этап обучения) и на C# (этап выполнения в приложении), сформирована тестовая выборка из 10 реальных тренировочных серий. Прогнозы модели LightGBM (Python) и среды выполнения ONNX Runtime (C#) совпали с медианным расхождением менее 0,001 кг.

## 6.4 Тестирование облачной синхронизации

Интеграционное тестирование синхронизации проводилось на двух устройствах (Windows и Android) с использованием одной учётной записи.

**Сценарий 1: базовая синхронизация.** Тренировка создана на устройстве А, выполнена синхронизация с устройства А, затем с устройства Б. Тренировка появилась на устройстве Б — тест пройден.

**Сценарий 2: конфликт одновременных изменений.** На устройствах А и Б в автономном режиме изменено название одной тренировки на разные значения. Сначала синхронизацию запустило устройство А (с более ранней меткой `UpdatedAt`), затем устройство Б (с более поздней меткой). После завершения обоих циклов на обоих устройствах закрепилось значение, введённое на устройстве Б, — стратегия «побеждает последняя запись» отработала корректно.

**Сценарий 3: синхронизация мягкого удаления.** Тренировка удалена на устройстве А, после синхронизации скрылась на устройстве Б. На обоих устройствах запись присутствует в БД с `IsDeleted = true`.

**Сценарий 4: смена аккаунта.** После выхода из аккаунта и входа под другой учётной записью кнопка «Полная пересинхронизация» корректно сбросила курсор `LastSyncUtc` и загрузила тренировки нового пользователя.

Сводные результаты интеграционного тестирования синхронизации приведены в таблице 6.3.

**Таблица 6.3 — Результаты интеграционного тестирования синхронизации**

| Сценарий | Описание | Статус |
|---|---|---|
| 1. Базовая синхронизация | Тренировка создана на устройстве А и передана на устройство Б через сервер | Пройден |
| 2. Конфликт одновременных изменений | Два устройства в автономном режиме изменяют одно поле — закрепляется значение с более поздним `UpdatedAt` | Пройден |
| 3. Синхронизация мягкого удаления | Тренировка удалена на устройстве А — скрылась на устройстве Б (`IsDeleted = true`) | Пройден |
| 4. Смена аккаунта | После повторного входа полная пересинхронизация загружает тренировки нового пользователя | Пройден |

## 6.5 Анализ производительности и пользовательского опыта

**Время загрузки страниц.** Среднее время загрузки `WorkoutListPage` с 50 тренировками составило 42 мс на Windows и 78 мс на Android. Требование ≤ 200 мс выполнено.

**Объём установочного пакета.** Установочный пакет для Android (APK): 38 МБ — требование ≤ 50 МБ выполнено. Основной вклад в объём вносят среда выполнения ONNX Runtime (≈ 12 МБ) и сама модель в формате ONNX (1,5 МБ).

**Оценка пользовательского опыта.** Проведено пилотное тестирование с участием 6 добровольцев — студентов, регулярно посещающих тренажёрный зал. По итогам тестирования собрана качественная обратная связь по неструктурированному опросу. Основные отмеченные достоинства: интуитивный ввод подходов, полезность подсказки «Через 28 дней: ~X кг», удобство применения рекомендации одной кнопкой. Выявленное замечание: отсутствие возможности просмотра истории конкретного упражнения в отдельном экране — отмечено как задача будущего развития.
