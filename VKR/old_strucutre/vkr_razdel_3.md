# 3. РАЗРАБОТКА КЛИЕНТСКОГО ПРИЛОЖЕНИЯ

## 3.1 Реализация паттерна MVVM

Клиентское приложение реализует архитектурный паттерн MVVM средствами библиотеки CommunityToolkit.Mvvm. Каждой странице интерфейса (представлению, описанному в XAML) соответствует класс модели представления, наследующий базовый тип `ObservableObject`. Состояние модели представления хранится в полях, помеченных атрибутом `[ObservableProperty]`, а пользовательские действия оформлены методами с атрибутом `[RelayCommand]`. На этапе компиляции генератор исходного кода преобразует такие поля в открытые свойства с уведомлением об изменении (`INotifyPropertyChanged`), а методы — в объекты команд (`ICommand`). Это избавляет разработчика от написания однотипного шаблонного кода и обеспечивает строгое разделение слоёв.

Связь представления и модели представления осуществляется через механизм привязки данных. Преобразование значений на границе слоёв выполняют классы-конвертеры, реализующие интерфейс `IValueConverter`: `WeightToHeightConverter` пересчитывает значение 1ПМ в высоту столбца графика, а `NullOrEmptyConverter` управляет видимостью элементов интерфейса при отсутствии данных. Модель представления не обращается к элементам интерфейса напрямую — единственным исключением являются диалоговые службы навигационной оболочки Shell, рассмотренные в п. 3.6.

Реакция на изменение состояния реализована через частичные методы вида `OnСвойствоChanged`, автоматически вызываемые генератором при изменении соответствующего свойства. Так, метод `OnFilterDateChanged` перестраивает список тренировок при смене даты фильтра, а метод `OnSheetRpeChanged` пересчитывает текстовую интерпретацию выбранной оценки RPE.

## 3.2 Структура навигации и основные экраны

Навигация в приложении построена на основе оболочки Shell с нижней панелью вкладок. Приложение содержит семь основных экранов:

*[Рисунок 3.1 — Схема навигации клиентского приложения FitApp — файл: `diagrams/diagram_3_1_navigation.drawio`]*

1. **WorkoutListPage** — список тренировок пользователя с датами и мышечными группами; точка входа для создания новых тренировок.
2. **WorkoutPage** — страница активной тренировки: добавление упражнений и подходов, ввод RPE, отображение подсказки модели прогнозирования («Через 28 дн.: ~152 кг») и рекомендации планировщика.
3. **ExercisePage** — каталог упражнений с фильтрацией по мышечным группам.
4. **ProgressPage** — страница статистики: тоннаж за период, тепловая карта активности, рекорды, график динамики 1ПМ.
5. **TemplatesPage / TemplateEditorPage** — создание и редактирование шаблонов тренировок.
6. **SettingsPage** — профиль пользователя (вес тела, возраст, пол, целевой RPE).
7. **AccountPage** — вход, регистрация и управление синхронизацией.

Переходы между экранами организованы по URI-маршрутам, зарегистрированным в навигационной оболочке Shell. При открытии страницы её модель представления выполняет асинхронную инициализацию: запрашивает необходимые данные из локальной базы данных, не блокируя поток интерфейса. Для модели представления тренировки применён приём с двумя конструкторами: первый используется для отображения списка тренировок, второй принимает конкретную тренировку и переводит экран в режим активной тренировки, загружая её упражнения, подходы и связанные мышечные группы.

## 3.3 Модуль ведения тренировок

Модуль ведения тренировок реализует основной пользовательский сценарий и сосредоточен в модели представления `WorkoutViewModel`.

### 3.3.1 Создание тренировки

Создание тренировки выполняется командой `AddWorkout`. Команда объединяет выбранные дату и время в единую метку, проверяет заполнение названия (при пустом значении пользователю выводится предупреждение) и сохраняет тренировку в локальной базе данных. Связь тренировки с мышечными группами реализована по схеме «многие ко многим»: для каждой выбранной группы создаётся запись-связка. При добавлении упражнения в тренировку его мышечные группы автоматически привязываются к тренировке методом `AttachExerciseMusclesToWorkoutAsync`, причём уже существующие связи пропускаются во избежание дублирования.

### 3.3.2 Ввод подхода

Ввод и редактирование подхода вынесены в выдвижную нижнюю панель, видимость которой управляется свойством `IsSetSheetOpen`. Панель содержит поля рабочего веса, числа повторений и оценки RPE, а также кнопки-инкременты (`AdjustWeight`, `AdjustReps`), изменяющие значение с фиксированным шагом. При открытии панели поля предварительно заполняются значениями последнего выполненного подхода данного упражнения, что ускоряет ввод однотипных данных. Сохранение (`SaveSheet`) немедленно записывает подход в локальную базу данных с проставлением метки `UpdatedAt = DateTime.UtcNow`. Такое решение обеспечивает сохранение без явного нажатия кнопки «Сохранить» и исключает потерю данных при закрытии приложения.

### 3.3.3 Оценка RPE и её интерпретация

Для ввода RPE применяется фиксированная шкала из восьми значений в диапазоне от 6 до 10, оформленных в виде набора кнопок-переключателей. При выборе значения частичный метод `OnSheetRpeChanged` пересчитывает текстовую интерпретацию нагрузки — краткий заголовок и пояснение (например, оценке 9 соответствует «Очень тяжело — могли бы сделать ещё 1 повторение»). Текстовая расшифровка помогает пользователю соотнести числовую оценку с субъективным ощущением и снижает разнобой в проставлении RPE.

### 3.3.4 Типы подходов

Каждый подход относится к одному из четырёх типов: разминочный, рабочий, выполненный до отказа и дроп-сет. Тип выбирается командой `ChangeSetType` через системное меню действий (`DisplayActionSheet`) и влияет на визуальное оформление подхода, а также учитывается при последующем анализе нагрузки.

### 3.3.5 Подсказка модели и применение плана

При загрузке упражнений тренировки (метод `LoadExercisesForWorkout`) для каждого упражнения асинхронно вычисляются прогноз 1ПМ (`OnnxPredictionService.PredictAsync`) и рекомендация на следующую тренировку (`WorkoutPlannerService.GetRecommendationAsync`). Запросы выполняются параллельно и ожидаются совместно через `Task.WhenAll`, что сокращает суммарное время загрузки страницы. Результаты отображаются вспомогательным текстом рядом с упражнением.

Команда «Применить план» (`ApplyPlan`) создаёт в базе рекомендованное число подходов с рассчитанными весом и числом повторений, оставляя поле RPE незаполненным для проставления пользователем по факту выполнения.

## 3.4 Модуль статистики и прогресса

Страница прогресса предоставляет несколько агрегированных представлений данных, вычисляемых SQL-запросами в классе `WorkoutDataBase`:

- **Тоннаж** (суммарный объём нагрузки за период, кг × повторения) — отображается по неделям в виде столбчатой диаграммы;
- **Личные рекорды** — максимальный 1ПМ по каждому упражнению за всё время;
- **Тепловая карта активности** — матрица «неделя × день недели», закрашенная пропорционально числу выполненных подходов (по аналогии с графиком активности GitHub);
- **График динамики 1ПМ** — ломаная по тренировкам с линией тренда (линейная регрессия по последним 5 точкам).

Тепловая карта активности реализована не средствами готовой библиотеки, а собственной отрисовкой на элементе `GraphicsView` через класс `ActivityCalendarDrawable`, реализующий интерфейс `IDrawable` из пространства имён Microsoft.Maui.Graphics. Класс рисует сетку из 12 недель по 7 дней: цвет ячейки определяется уровнем интенсивности тоннажа за день, слева размещаются подписи дней недели, сверху — подписи месяцев при их смене, текущий день выделяется обводкой, а в ячейках с активностью выводится число дня. Цветовая палитра адаптируется к тёмной теме оформления (свойство `IsDarkTheme`). Для обеспечения интерактивности класс сохраняет координаты прямоугольников всех ячеек (`CellRects`), что позволяет связанному коду сопоставить точку касания с конкретным днём.

Столбчатые диаграммы тоннажа строятся библиотекой LiveChartsCore.SkiaSharpView.Maui 2.0, поддерживающей аппаратное ускорение на Android и Windows. Значения 1ПМ при построении графика динамики масштабируются в высоту элементов конвертером `WeightToHeightConverter`.

## 3.5 Каталог упражнений и шаблоны

Каталог упражнений содержит более 50 упражнений, предварительно загруженных при первом запуске приложения. Каждое упражнение характеризуется типом оборудования, основными и вспомогательными мышечными группами и признаком многосуставности (`IsCompound`). Пользователь может добавлять собственные упражнения. Выбор упражнения для добавления в тренировку выполняется на отдельном экране с фильтрацией по мышечным группам (модель представления `ExerciseSelectionViewModel`).

Шаблоны тренировок позволяют сохранить набор упражнений с целевым числом подходов и повторений и использовать их для быстрого создания новой тренировки. Редактирование шаблона выполняется на странице `TemplateEditorPage` с поддержкой переупорядочивания упражнений перетаскиванием.

## 3.6 Обработка ошибок и взаимодействие с пользователем

Диалоговое взаимодействие реализовано через службы навигационной оболочки Shell, вызываемые непосредственно из команд моделей представления. Все необратимые операции — удаление тренировки, упражнения или подхода — предваряются запросом подтверждения (`Shell.Current.DisplayAlert`) с явным указанием последствий. Ввод пользователя проверяется на корректность: например, попытка создать тренировку без названия пресекается предупреждающим сообщением. Выбор одного варианта из нескольких (в частности, типа подхода) выполняется через системное меню действий (`DisplayActionSheet`).

Поскольку приложение придерживается принципа приоритета локальных данных, ни одна из перечисленных операций не зависит от наличия сетевого подключения. Ошибки сетевого взаимодействия возникают исключительно при синхронизации с сервером и обрабатываются отдельно.

## 3.7 Облачная синхронизация

До реализации данного компонента приложение функционировало исключительно в автономном режиме, а данные хранились только на устройстве. Это порождало два ограничения: потерю данных при переустановке или смене устройства и невозможность вести тренировочный журнал на нескольких устройствах одновременно. Облачная синхронизация устраняет эти ограничения, не нарушая принципа приоритета автономной работы: отсутствие сети не блокирует ни одну функцию приложения, а синхронизация выступает дополнительной возможностью, а не условием работы.

### 3.7.1 Архитектура серверной части

Рассматривались два варианта реализации сервера. Первый — полная реализация на ASP.NET Core с Entity Framework Core, таблицами сессий, токенами обновления и управлением версиями записей (оценочная трудоёмкость около 16 часов). Второй, выбранный, — минимальный API на ASP.NET Core 9 без объектно-реляционного отображения, с прямыми SQL-запросами через Npgsql, тремя конечными точками и аутентификацией по JWT без токенов обновления. Выбор обоснован принципом разумной достаточности для задач работы и необходимостью оставить достаточное время на тестирование и отладку.

Серверная часть размещена в Docker-контейнере с двухстадийной сборкой (multi-stage build): стадия build на образе `dotnet/sdk:9.0` с последующим копированием артефактов в минимальный образ `dotnet/aspnet:9.0`, что уменьшает итоговый образ примерно в четыре раза. Конфигурация задаётся через переменные окружения (`DATABASE_URL`, `Jwt__Secret`) в соответствии с принципами двенадцатифакторной методологии (12 Factor App) [27].

### 3.7.2 Глобальные идентификаторы и разрешение конфликтов

Локальный первичный ключ `Id` генерируется автоинкрементом и не уникален между устройствами. Для устранения коллизий каждая синхронизируемая сущность получает поле `SyncId` — строку с глобальным идентификатором (GUID), генерируемым клиентом при создании записи; на сервере `SyncId` является первичным ключом. Связи между сущностями дублируются парой полей: целочисленный внешний ключ для локальных соединений и строковый `*SyncId` для серверной стороны. Для ранее созданных записей, не имевших идентификатора, выполняется одноразовая миграция `BackfillSyncIdsAsync`.

Разрешение конфликтов реализовано по стратегии «побеждает последняя запись» на основе поля `UpdatedAtUtc`. Серверная операция вставки-или-обновления при совпадении `SyncId` выполняет обновление существующей строки только в том случае, если метка времени поступившей записи позднее уже сохранённой; в противном случае изменение отклоняется. Тем самым более старые данные не могут перезаписать более новые, а порядок поступления пакетов от устройств не влияет на итоговое состояние. Такое поведение обеспечивается конструкцией `INSERT ... ON CONFLICT DO UPDATE` с дополнительным условием по метке времени, поддерживаемой PostgreSQL [22]; полное определение таблиц и соответствующих запросов приведено в Приложении Б. Удаление реализовано как мягкое (флаг `IsDeleted`), что обеспечивает корректное распространение факта удаления между устройствами.

### 3.7.4 Развёртывание

В первоначальной реализации серверным хранилищем служил SQLite с файлом в директории `/tmp`. На платформе Render.com эта директория эфемерна: при каждом перезапуске контейнера, в том числе после холодного запуска при пробуждении из спящего режима, её содержимое уничтожается, что приводило к полной потере данных и делало межустройственную синхронизацию неработоспособной. Хранилище заменено на PostgreSQL, предоставляемый платформой Neon (serverless PostgreSQL, бесплатный тариф — 512 МБ постоянного хранилища); данные сохраняются независимо от жизненного цикла контейнера. Переход потребовал адаптации синтаксиса запросов (параметры `$name` → `@name`, получение вставленного идентификатора `last_insert_rowid()` → `RETURNING Id`) и корректного разбора строки подключения Neon в URI-формате через `NpgsqlConnectionStringBuilder`; реализация класса `AppDb.BuildConnectionString` приведена в Приложении А.

Сервис развёрнут на платформе Render.com по бесплатному тарифу с автоматическим развёртыванием при фиксации изменений в репозитории GitHub; сертификат TLS управляется платформой. Ограничение тарифа — засыпание сервиса после 15 минут неактивности и задержка до 30 секунд на первый запрос после сна. В `AccountViewModel` реализован отложенный индикатор: если операция длится дольше 5 секунд, строка статуса меняется на «Сервер засыпал, поднимаем (~30 сек)...», а тайм-аут `HttpClient` увеличен до 60 секунд.
