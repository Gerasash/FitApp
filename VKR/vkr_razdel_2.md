# 2. ПРОЕКТИРОВАНИЕ ИНФОРМАЦИОННОЙ СИСТЕМЫ

## 2.1 Выбор технологического стека

**Клиентская часть.** Требование кроссплатформенности (Android и Windows) при сохранении нативного пользовательского опыта определило выбор платформы **.NET MAUI 9** (Multi-platform App UI). MAUI позволяет разрабатывать приложения для Android, iOS, Windows и macOS из единой кодовой базы на языке C#, компилируя нативные приложения для каждой платформы [19]. Альтернативой рассматривался Flutter (Google), однако экосистема .NET обеспечивает более тесную интеграцию с библиотеками машинного обучения (Microsoft.ML, ONNX Runtime) и упрощает разработку серверной части на единой платформе.

Для реализации паттерна MVVM применяется библиотека **CommunityToolkit.Mvvm 8.4** [20], обеспечивающая генерацию кода через атрибуты `[ObservableProperty]` и `[RelayCommand]` при компиляции, что исключает написание шаблонного кода.

Локальная база данных — **SQLite** через библиотеку **sqlite-net-pcl 1.9** [21]. SQLite обеспечивает надёжное хранение данных на устройстве без необходимости установки сервера, поддерживает транзакции и индексы, работает одинаково на Android и Windows.

**Серверная часть.** Сервер реализован на **ASP.NET Core 9** — высокопроизводительном кроссплатформенном фреймворке для веб-приложений. Выбор обусловлен единством языка (C#) с клиентом, что упрощает совместное использование DTO-классов и снижает вероятность рассогласования схем.

База данных сервера — **PostgreSQL** через библиотеку **Npgsql**. PostgreSQL обеспечивает постоянное хранение данных в облачной среде, поддерживает конструкцию `INSERT ... ON CONFLICT DO UPDATE WHERE`, необходимую для реализации LWW-синхронизации, и хорошо интегрируется с платформами облачного хостинга [22].

**ML-инфраструктура.** Модель обучается на Python 3.11 с использованием LightGBM, конвертируется в ONNX и исполняется в приложении через **Microsoft.ML.OnnxRuntime 1.20.1** — кроссплатформенную C/C++-библиотеку с C#-оберткой, работающей нативно на Android и Windows.

## 2.2 Архитектура клиентского приложения

Клиентское приложение построено по архитектурному паттерну **«Модель–Вид–Модель представления» (MVVM)**. Паттерн обеспечивает разделение ответственности между слоями: слой модели отвечает за данные и бизнес-логику, слой вида — за отображение, а модель представления связывает их через механизм привязки данных [23].

**Структура проекта:**

- `Models/` — сущности SQLite: `Workout`, `WorkoutExercise`, `ExerciseSet`, `Exercise`, `MuscleGroup`, `UserProfile`;
- `ViewModels/` — частичные классы, наследующие `ObservableObject`; команды и наблюдаемые свойства генерируются на этапе компиляции через атрибуты CommunityToolkit;
- `Views/` — XAML-страницы и файлы вспомогательного кода;
- `Services/` — сервисы бизнес-логики: предсказания ONNX, планировщик, синхронизация, клиент аутентификации;
- `Data/` — единственный класс доступа к базе данных `WorkoutDataBase`.

Навигация управляется через механизм маршрутизации **Shell**, встроенный в MAUI: переходы между страницами задаются URI-адресами через вызов `Shell.Current.GoToAsync`.

**Принцип «локальные данные прежде всего».** Все операции чтения и записи выполняются с локальной базой SQLite. Синхронизация с сервером инициируется исключительно по явному действию пользователя (нажатие кнопки «Синхронизировать»). При отсутствии сетевого подключения ни одна функция приложения не блокируется.

**Внедрение зависимостей.** В точке входа `MauiProgram.cs` регистрируются все сервисы и страницы. `WorkoutDataBase` зарегистрирован как одиночка (одно соединение на весь жизненный цикл приложения), прочие компоненты создаются при каждом запросе.

## 2.3 Проектирование базы данных

Локальная база данных содержит семь основных таблиц. Схема отражает физическую модель данных, применяемую в `WorkoutDataBase.cs` (листинг схемы приведён в Приложении А).

*[Рисунок 2.2 — ER-диаграмма базы данных клиента FitApp — файл: `diagrams/diagram_2_2_er.drawio`]*

Ключевые решения схемы:

1. **Мягкое удаление** (`IsDeleted`): вместо физического удаления строк устанавливается флаг. Это обеспечивает корректное распространение факта удаления при синхронизации — удалённая запись с новым `UpdatedAt` «перезапишет» существующую запись на других устройствах по стратегии LWW.

2. **Поле SyncId**: каждая синхронизируемая сущность (`Workout`, `WorkoutExercise`, `ExerciseSet`) имеет поле `SyncId` — строку с GUID, генерируемым клиентом при создании записи. Это поле является первичным ключом на сервере и обеспечивает однозначную идентификацию записей между устройствами без коллизий автоинкрементных идентификаторов.

3. **Временны́е метки**: поле `UpdatedAt` (тип `DateTime`) проставляется клиентом при каждом изменении записи через `DateTime.UtcNow`. Используется фильтром выборки при построении push-батча синхронизации.

4. **Связь Workout ↔ MuscleGroup**: реализована через таблицу-связку `WorkoutMuscleGroups` (отношение многие-ко-многим). При обновлении тренировки старые связи удаляются и заново вставляются.

**Серверная схема.** На стороне сервера поддерживаются четыре таблицы: `Users`, `Workouts`, `WorkoutExercises`, `ExerciseSets`. Первичным ключом таблиц тренировочных данных является `SyncId`. Схема сервера приведена в Приложении А.

## 2.4 Проектирование серверной части

Сервер реализован в виде минимального REST API на ASP.NET Core 9 с тремя эндпоинтами:

**Таблица 2.1 — Эндпоинты API сервера FitApp**

| Метод | URL | Аутентификация | Описание |
|---|---|---|---|
| POST | `/auth/register` | Нет | Регистрация нового пользователя по логину и паролю; возвращает JWT-токен (30 дней) |
| POST | `/auth/login` | Нет | Аутентификация пользователя; возвращает JWT-токен (30 дней) |
| POST | `/sync` | Bearer JWT | Двунаправленная синхронизация: принимает push-батч изменений, возвращает pull-батч изменений сервера |

**Аутентификация** реализована через JWT-токены (JSON Web Token, RFC 7519 [24]) со схемой подписи HS256. Токен содержит идентификатор пользователя и срок действия (30 дней). Хеширование паролей выполняется алгоритмом BCrypt с ценой вычисления (work factor) 10 [25].

**Эндпоинт синхронизации** `POST /sync` принимает тело запроса, содержащее три массива изменённых сущностей (`Workouts`, `WorkoutExercises`, `ExerciseSets`) и метку `LastSyncUtc`. В ответ возвращаются все сущности пользователя, изменённые после `LastSyncUtc`, и серверное время `ServerTimeUtc`. Детальное описание протокола синхронизации приведено в разделе 5.
