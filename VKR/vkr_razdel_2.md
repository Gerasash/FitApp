# 2. ПРОЕКТИРОВАНИЕ ИНФОРМАЦИОННОЙ СИСТЕМЫ

## 2.1 Выбор технологического стека

Клиентское приложение разрабатывается на платформе **.NET MAUI 9** (Multi-platform App UI). Её выбор продиктован требованием кроссплатформенности (Android и Windows) при сохранении платформенного быстродействия и внешнего вида интерфейса. Платформа .NET MAUI позволяет создавать приложения для Android, iOS, Windows и macOS из единой кодовой базы на языке C#, компилируя для каждой целевой системы собственное платформенное приложение [19]. В качестве альтернативы рассматривался каркас Flutter, однако экосистема .NET обеспечивает более тесную интеграцию с библиотеками машинного обучения (Microsoft.ML, ONNX Runtime) и позволяет вести разработку серверной части на том же языке.

Паттерн MVVM реализуется средствами библиотеки **CommunityToolkit.Mvvm 8.4** [20]: она генерирует код на этапе компиляции по атрибутам `[ObservableProperty]` и `[RelayCommand]`, избавляя разработчика от написания однотипного шаблонного кода.

В качестве локальной базы данных выбрана **SQLite**, доступ к которой осуществляется через библиотеку **sqlite-net-pcl 1.9** [21]. Эта СУБД обеспечивает надёжное хранение данных непосредственно на устройстве, не требует установки отдельного сервера, поддерживает транзакции и индексы и работает одинаково на Android и Windows.

Серверная часть реализована на программной платформе **ASP.NET Core 9** — высокопроизводительном кроссплатформенном каркасе для веб-приложений. Его выбор обусловлен единством языка программирования (C#) с клиентом, что упрощает совместное использование классов передачи данных (DTO) и снижает риск рассогласования схем между клиентом и сервером.

Базой данных сервера служит **PostgreSQL**, доступ к которой выполняется через библиотеку **Npgsql**. Эта СУБД обеспечивает постоянное хранение данных в облачной среде, поддерживает конструкцию `INSERT ... ON CONFLICT DO UPDATE WHERE`, необходимую для синхронизации по стратегии «выигрывает последняя запись» (LWW, Last-Write-Wins), и хорошо интегрируется с облачными платформами размещения приложений [22].

Инфраструктура машинного обучения организована следующим образом. Модель обучается на языке Python 3.11 средствами LightGBM, преобразуется в формат ONNX и исполняется в приложении посредством библиотеки **Microsoft.ML.OnnxRuntime 1.20.1** — кроссплатформенной библиотеки на C/C++ с оболочкой для C#, работающей на Android и Windows без промежуточных слоёв.

## 2.2 Архитектура клиентского приложения

Клиентское приложение построено по архитектурному паттерну **«модель — представление — модель представления» (MVVM)**. Паттерн разграничивает ответственность между слоями: модель отвечает за данные и бизнес-логику, представление — за отображение, а модель представления связывает их посредством механизма привязки данных [23].

Структура проекта отражена в организации каталогов решения:

- `Models/` — сущности SQLite: `Workout`, `WorkoutExercise`, `ExerciseSet`, `Exercise`, `MuscleGroup`, `UserProfile`;
- `ViewModels/` — частичные классы, наследующие `ObservableObject`; команды и наблюдаемые свойства генерируются на этапе компиляции через атрибуты CommunityToolkit;
- `Views/` — XAML-страницы и файлы вспомогательного кода;
- `Services/` — сервисы бизнес-логики: предсказания ONNX, планировщик, синхронизация, клиент аутентификации;
- `Data/` — единственный класс доступа к базе данных `WorkoutDataBase`.

Навигация управляется встроенным в .NET MAUI механизмом маршрутизации **Shell**: переходы между страницами задаются URI-адресами через вызов `Shell.Current.GoToAsync`.

Приложение придерживается принципа приоритета локальных данных. Все операции чтения и записи выполняются с локальной базой SQLite, а синхронизация с сервером запускается исключительно по явному действию пользователя — нажатию кнопки «Синхронизировать». При отсутствии сетевого подключения ни одна функция приложения не блокируется.

Внедрение зависимостей настраивается в точке входа `MauiProgram.cs`, где регистрируются все сервисы и страницы. Класс `WorkoutDataBase` зарегистрирован как одиночка (singleton) — единственное соединение на весь жизненный цикл приложения, тогда как остальные компоненты создаются заново при каждом запросе.

## 2.3 Проектирование базы данных

Локальная база данных содержит семь основных таблиц. Схема отражает физическую модель данных, применяемую в `WorkoutDataBase.cs` (листинг схемы приведён в Приложении А).

*[Рисунок 2.2 — ER-диаграмма базы данных клиента FitApp — файл: `diagrams/diagram_2_2_er.drawio`]*

Ключевые решения схемы:

1. **Мягкое удаление** (`IsDeleted`): вместо физического удаления строк устанавливается флаг. Это обеспечивает корректное распространение факта удаления при синхронизации — удалённая запись с новым `UpdatedAt` «перезапишет» существующую запись на других устройствах по стратегии LWW.

2. **Поле SyncId**: каждая синхронизируемая сущность (`Workout`, `WorkoutExercise`, `ExerciseSet`) имеет поле `SyncId` — строку с GUID, генерируемым клиентом при создании записи. Это поле является первичным ключом на сервере и обеспечивает однозначную идентификацию записей между устройствами без коллизий автоинкрементных идентификаторов.

3. **Временны́е метки**: поле `UpdatedAt` (тип `DateTime`) проставляется клиентом при каждом изменении записи через `DateTime.UtcNow`. Оно служит фильтром выборки при формировании пакета исходящих изменений во время синхронизации.

4. **Связь Workout ↔ MuscleGroup**: реализована через таблицу-связку `WorkoutMuscleGroups` (отношение многие-ко-многим). При обновлении тренировки старые связи удаляются и заново вставляются.

На стороне сервера поддерживаются четыре таблицы: `Users`, `Workouts`, `WorkoutExercises`, `ExerciseSets`. Первичным ключом таблиц тренировочных данных служит `SyncId`. Схема сервера приведена в Приложении А.

## 2.4 Проектирование серверной части

Сервер реализован в виде минимального REST API на ASP.NET Core 9 и предоставляет три конечные точки.

**Таблица 2.1 — Эндпоинты API сервера FitApp**

| Метод | URL | Аутентификация | Описание |
|---|---|---|---|
| POST | `/auth/register` | Нет | Регистрация нового пользователя по логину и паролю; возвращает JWT-токен (30 дней) |
| POST | `/auth/login` | Нет | Аутентификация пользователя; возвращает JWT-токен (30 дней) |
| POST | `/sync` | Bearer JWT | Двунаправленная синхронизация: принимает пакет исходящих изменений клиента, возвращает пакет входящих изменений сервера |

Аутентификация реализована с помощью токенов JWT (JSON Web Token, RFC 7519 [24]) со схемой подписи HS256. Токен содержит идентификатор пользователя и срок действия (30 дней). Пароли хранятся в виде хешей, вычисляемых алгоритмом BCrypt с ценой вычисления (work factor), равной 10 [25].

Конечная точка синхронизации `POST /sync` принимает тело запроса с тремя массивами изменённых сущностей (`Workouts`, `WorkoutExercises`, `ExerciseSets`) и меткой времени `LastSyncUtc`. В ответ сервер возвращает все сущности пользователя, изменённые после момента `LastSyncUtc`, а также собственное текущее время `ServerTimeUtc`. Подробное описание протокола синхронизации приведено в разделе 5.
