# 5. РАЗРАБОТКА ОБЛАЧНОЙ СИНХРОНИЗАЦИИ

## 5.1 Постановка задачи и обоснование offline-first

К моменту начала разработки данного компонента приложение функционировало исключительно в автономном режиме. Хранение данных только на устройстве порождало два ключевых ограничения: потерю данных при переустановке или смене устройства, а также невозможность ведения тренировочного журнала на нескольких устройствах одновременно.

Цель данного этапа — реализация облачной синхронизации при сохранении офлайн-первого свойства: отсутствие сети не должно блокировать ни одну функцию приложения. Синхронизация выступает дополнительной возможностью, а не необходимым условием работы.

## 5.2 Архитектура серверной части

Рассматривались два варианта реализации сервера.

**Полная реализация** — ASP.NET Core с Entity Framework Core, таблицами сессий, refresh-токенами и управлением версиями записей. Оценочная трудоёмкость составила около 16 часов.

**Минимальный API** (выбранный вариант) — ASP.NET Core 9 без ORM, прямые SQL-запросы через Npgsql, три эндпоинта, JWT без refresh. Выбор обоснован принципом разумной достаточности для задач ВКР и необходимостью обеспечить достаточное время на тестирование и отладку.

Серверная часть размещена в Docker-контейнере с двухстадийной сборкой (multi-stage build): стадия `build` на образе `dotnet/sdk:9.0` с последующим копированием артефактов в минимальный образ `dotnet/aspnet:9.0`. Такой подход уменьшает итоговый образ примерно в четыре раза.

Конфигурация задаётся через переменные окружения (`DATABASE_URL`, `Jwt__Secret`), что соответствует принципам двенадцатифакторной методологии (12 Factor App) [27].

## 5.3 Протокол синхронизации

### 5.3.1 Глобальные идентификаторы

Локальный первичный ключ `Id` генерируется автоинкрементом и не уникален между устройствами. Для устранения коллизий каждая синхронизируемая сущность получает поле `SyncId` — строку с GUID, генерируемым клиентом при создании записи. На сервере `SyncId` является первичным ключом.

Связи между сущностями дублированы парой полей: целочисленный FK (для локальных JOIN) и строковый `*SyncId` (для серверной стороны). Для ранее созданных записей, не имевших GUID, выполняется одноразовая миграция `BackfillSyncIdsAsync`.

### 5.3.2 Стратегия Last-Write-Wins

Разрешение конфликтов реализовано по стратегии **Last-Write-Wins** (LWW) по полю `UpdatedAtUtc`. Серверная операция вставки-или-обновления обеспечивает, что более старые данные не перезаписывают более новые:

```sql
INSERT INTO Workouts (...) VALUES (...)
ON CONFLICT(SyncId) DO UPDATE SET ...
WHERE excluded.UpdatedAtUtc > Workouts.UpdatedAtUtc;
```

*(Полное SQL-определение таблиц и UPSERT-запросов приведено в Приложении Б.)*

Данная конструкция поддерживается PostgreSQL начиная с версии 9.5 без изменений [22]. Удаление реализовано как мягкое (флаг `IsDeleted`) — это обеспечивает корректное распространение факта удаления между устройствами.

JWT хранится через `Microsoft.Maui.Storage.SecureStorage`. На Android используется `EncryptedSharedPreferences`, на Windows — DPAPI. Это обеспечивает платформенный уровень защиты учётных данных.

## 5.4 Развёртывание

### 5.4.1 База данных: переход на PostgreSQL

В первоначальной реализации в качестве серверного хранилища использовался SQLite с файлом в директории `/tmp`. На платформе Render.com директория `/tmp` является эфемерной: при каждом перезапуске контейнера (в том числе после «холодного старта» при пробуждении из спящего режима) содержимое директории уничтожается. Это приводило к полной потере данных при каждом перезапуске, что делало межустройственную синхронизацию неработоспособной.

Для устранения проблемы хранилище заменено на **PostgreSQL**, предоставляемый платформой **Neon** (serverless PostgreSQL, бесплатный тариф — 512 МБ постоянного хранилища). Данные сохраняются независимо от жизненного цикла контейнера.

Потребовалась адаптация синтаксиса: параметры запросов (`$name` → `@name`), получение вставленного идентификатора (`last_insert_rowid()` → `RETURNING Id`), а также корректный парсинг URI-формата строки подключения Neon через `NpgsqlConnectionStringBuilder`. Реализация класса `AppDb.BuildConnectionString` описана в Приложении А.

### 5.4.2 Хостинг

Сервис развёрнут на платформе **Render.com** по тарифу Free с автоматическим деплоем при коммите в GitHub (CI/CD). Сертификат TLS управляется платформой автоматически.

Ограничение бесплатного тарифа — сервис засыпает после 15 минут неактивности, первый запрос после сна занимает до 30 секунд. В `AccountViewModel` реализован отложенный индикатор: если операция выполняется дольше 5 секунд, строка статуса меняется на «Сервер засыпал, поднимаем (~30 сек)...». Таймаут `HttpClient` увеличен до 60 секунд.

## 5.5 Проблема clock skew и решения

В ходе интеграционного тестирования была обнаружена нетривиальная проблема, характерная для распределённых систем с несколькими временными шкалами.

**Проблема.** Курсор `LastSyncUtc` первоначально сохранялся как `ServerTimeUtc` — серверное время момента обработки запроса. Метки `UpdatedAt` у локальных записей проставляются клиентом через `DateTime.UtcNow`. При расхождении системных часов клиента и сервера (в частности, при неверно настроенном часовом поясе Windows, когда локальное время подаётся как UTC) возникает ситуация `ServerTimeUtc > client.UtcNow`. Следствие: условие `UpdatedAt >= LastSyncUtc` ложно для всех локальных записей — push-батч всегда пуст, тренировки не синхронизируются.

Диагностический лог подтвердил расхождение в три часа: `max UpdatedAt = 19:38:59`, `LastSyncUtc = 22:31:23`, `client.UtcNow = 19:39:17`.

**Принятые решения:**

1. **Курсор в клиентской временной шкале.** `LastSyncUtc` сохраняется как `clientNowAtStart = DateTime.UtcNow`, фиксированное до отправки запроса. Поскольку `UpdatedAt` у всех локальных записей выставляется той же шкалой, фильтр `UpdatedAt >= since` математически корректен независимо от расхождения часов.

2. **Автосброс битого курсора.** При `LastSyncUtc > clientNowAtStart` (курсор оказался в будущем) курсор сбрасывается в `null`, и следующая синхронизация отправляет все записи.

3. **Нестрогое неравенство** `>=` вместо `>` перекрывает расхождения порядка единиц миллисекунд. Идемпотентность сервера гарантирует безопасность повторной отправки.

4. **Кнопка принудительной пересинхронизации** в интерфейсе страницы «Аккаунт» для ручного восстановления в нештатных ситуациях.

Данный опыт иллюстрирует фундаментальное требование к распределённым системам: нельзя смешивать временны́е метки из разных источников при вычислении одного предиката [28].
