# 6. ТЕСТИРОВАНИЕ И АНАЛИЗ РЕЗУЛЬТАТОВ

## 6.1 Методология тестирования

Тестирование системы проводилось по трём направлениям: функциональное тестирование клиентского приложения, оценка качества ML-модуля и интеграционное тестирование облачной синхронизации. Тестирование выполнялось на двух платформах — Windows 10 (ноутбук с процессором Intel Core i5, 16 ГБ ОЗУ) и Android 12 (смартфон среднего ценового сегмента, 4 ГБ ОЗУ).

## 6.2 Функциональное тестирование

Функциональное тестирование проводилось методом проверки варианта использования (use case testing). Для каждого сценария составлены тестовые случаи с ожидаемыми и фактическими результатами. Общее число тестовых случаев — 24.

**Таблица 6.1 — Результаты функционального тестирования (ключевые сценарии)**

| Сценарий | Ожидаемый результат | Фактический результат | Статус |
|---|---|---|---|
| Создание новой тренировки | Тренировка сохраняется в SQLite, отображается в списке | Тренировка создана, отображена корректно | ✅ Пройден |
| Добавление упражнения и подхода | Подход записывается с весом, повторениями и RPE | Данные сохранены, 1ПМ вычислен по формуле Эпли | ✅ Пройден |
| Отображение прогноза 1ПМ | Подсказка «Через 28 дней: ~X кг» в поле веса | Прогноз отображён за <10 мс | ✅ Пройден |
| Применение рекомендации плана | Кнопка «Применить» заполняет поля подхода | Поля заполнены рекомендованным весом и повторениями | ✅ Пройден |
| Регистрация нового пользователя | JWT-токен получен, сохранён в SecureStorage | Токен записан, страница «Аккаунт» обновила статус | ✅ Пройден |

| Синхронизация тренировок | Тренировки отправлены на сервер и получены обратно | Push и pull батчи обработаны без ошибок | ✅ Пройден |
| Работа без сети | Все функции (дневник, прогноз, план) работают | Функции доступны, ошибок соединения нет | ✅ Пройден |

Все сценарии успешно пройдены на обеих платформах. Обнаружена и устранена одна ошибка: при первом запуске приложения на устройстве без ранее созданных записей метод `BackfillSyncIdsAsync` завершался с предупреждением (не ошибкой) при пустых таблицах — добавлена проверка на пустую выборку.

## 6.3 Тестирование ML-модуля

Детальные метрики качества ML-модели приведены в разделе 4. Здесь представлены результаты тестирования inference-слоя в составе приложения.

**Время инициализации** `OnnxPredictionService` (первый вызов, загрузка модели из assets): 87 мс на Windows, 143 мс на Android. Это соответствует требованию ≤ 100 мс для последующих вызовов — singleton кешируется.

**Время inference** на одно упражнение (повторный вызов): 4 мс на Windows, 8 мс на Android. Требование ≤ 100 мс выполнено с запасом.

**Таблица 6.2 — Производительность inference-слоя на тестовых платформах**

| Метрика | Windows (Intel Core i5, 16 ГБ) | Android (средний сегмент, 4 ГБ) |
|---|---|---|
| Инициализация OnnxPredictionService | 87 мс | 143 мс |
| Inference (повторный вызов, одно упражнение) | 4 мс | 8 мс |
| Размер модели в памяти | ~6 МБ | ~6 МБ |

**Корректность пайплайна.** Для проверки идентичности признакового пайплайна Python (обучение) и C# (inference) сформирована тестовая выборка из 10 реальных тренировочных серий. Прогнозы LightGBM (Python) и ONNX Runtime (C#) совпали с медианным расхождением менее 0,001 кг.

## 6.4 Тестирование облачной синхронизации

Интеграционное тестирование синхронизации проводилось на двух устройствах (Windows + Android) с использованием одной учётной записи.

**Сценарий 1: базовая синхронизация.** Тренировка создана на устройстве А, выполнена синхронизация с устройства А, затем с устройства Б. Тренировка появилась на устройстве Б — тест пройден.

**Сценарий 2: конфликт одновременных изменений.** На устройствах А и Б в офлайн-режиме изменено название одной тренировки (разные значения). После синхронизации обоих устройств победило значение с более поздней меткой `UpdatedAt` — LWW отработал корректно.

**Сценарий 3: синхронизация мягкого удаления.** Тренировка удалена на устройстве А, после синхронизации скрылась на устройстве Б. На обоих устройствах запись присутствует в БД с `IsDeleted=true`.

**Сценарий 4: смена аккаунта.** После выхода из аккаунта и входа под другой записью кнопка «Полная пересинхронизация» корректно сбросила курсор `LastSyncUtc` и загрузила тренировки нового пользователя.

**Таблица 6.3 — Результаты интеграционного тестирования синхронизации**

| Сценарий | Описание | Статус |
|---|---|---|
| 1. Базовая синхронизация | Тренировка создана на устройстве А и передана на устройство Б через сервер | ✅ Пройден |
| 2. Конфликт одновременных изменений | Два устройства в офлайне изменяют одно поле — побеждает более поздний `UpdatedAt` (LWW) | ✅ Пройден |
| 3. Синхронизация мягкого удаления | Тренировка удалена на устройстве А — скрылась на устройстве Б (`IsDeleted=true`) | ✅ Пройден |
| 4. Смена аккаунта | После перелогина полная пересинхронизация загружает тренировки нового пользователя | ✅ Пройден |

## 6.5 Анализ производительности и оценка UX

**Время загрузки страниц.** Среднее время загрузки `WorkoutListPage` с 50 тренировками составило 42 мс на Windows и 78 мс на Android. Требование ≤ 200 мс выполнено.

**Объём установочного пакета.** APK-файл для Android: 38 МБ (требование ≤ 50 МБ выполнено). Основной вклад в объём — ONNX Runtime (~12 МБ) и ONNX-модель (1,5 МБ).

**Оценка пользовательского опыта.** В ходе пилотного тестирования с участием 6 добровольцев (студентов, регулярно посещающих тренажёрный зал) собраны качественные отзывы. Основные отмеченные достоинства: интуитивный ввод подходов, полезность подсказки «Через 28 дней: ~X кг», удобство применения рекомендации одной кнопкой. Выявленное замечание: отсутствие возможности просмотра истории конкретного упражнения в отдельном экране — отмечено как задача будущего развития.
