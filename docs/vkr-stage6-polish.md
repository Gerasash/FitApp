# Этап 6. Облачная синхронизация и улучшение пользовательского опыта

Текст для включения в пояснительную записку ВКР. Описывает архитектуру серверной части, протокол синхронизации, развёртывание в облаке Render, а также пакет UX-улучшений, выполненных перед защитой.

---

## 6.1. Постановка задачи

К моменту начала шестого этапа разработки приложение представляло собой автономный клиент на базе .NET MAUI с локальной базой данных SQLite. Тренировки, шаблоны и история сетов сохранялись только на устройстве пользователя, что порождало два ограничения:

1. **Потеря данных** при переустановке приложения или смене смартфона.
2. **Невозможность использования с нескольких устройств** — атлет не мог одновременно записывать тренировки на телефоне и просматривать прогресс, например, на планшете.

Цель этапа — спроектировать и реализовать механизм облачной синхронизации, сохранив автономность приложения (offline-first): отсутствие соединения с сервером не должно блокировать ни одну функцию.

## 6.2. Выбор архитектуры

Рассматривались два альтернативных подхода.

**Полная реализация** предполагала использование ASP.NET Core с Entity Framework Core, отдельной таблицей сессий, refresh-токенов и полноценным контролем версий записей. Оценочная трудоёмкость — около 16 часов.

**Минимальный API (выбранный вариант).** Сервер реализован на ASP.NET Core 9 без ORM: прямые SQL-запросы к SQLite через `Microsoft.Data.Sqlite`. Авторизация — JWT-токены HS256 со сроком действия 30 дней, без механизма обновления. Хеширование паролей — BCrypt (`BCrypt.Net-Next`). Реализовано всего три эндпоинта:

- `POST /auth/register` — регистрация и выдача токена;
- `POST /auth/login` — вход и выдача токена;
- `POST /sync` — двунаправленная синхронизация (защищённый эндпоинт).

Выбор обоснован принципом разумной достаточности: для ВКР важна работоспособность системы и демонстрация архитектурных решений, а не индустриальная зрелость кода. Сокращение объёма позволило выделить время на качество клиентской части.

## 6.3. Протокол синхронизации

### 6.3.1. Глобальные идентификаторы

Локальный первичный ключ записи (целочисленный `Id`) генерируется автоинкрементом и поэтому не уникален между устройствами. Для решения проблемы коллизий введён дополнительный столбец `SyncId` — строка с GUID, генерируемая на клиенте при создании записи:

```csharp
[Indexed(Unique = true)]
public string SyncId { get; set; } = Guid.NewGuid().ToString();
```

На сервере именно `SyncId` выступает первичным ключом таблиц. Локальный `Id` остаётся как технический идентификатор внутри устройства. Связи между сущностями (`WorkoutExercise → Workout`, `ExerciseSet → WorkoutExercise`) дублированы парой полей: целочисленный FK (для локальных JOIN) и `*SyncId` (для серверной стороны).

Для уже существующих локальных записей выполняется одноразовая миграция `BackfillSyncIdsAsync`, которая проставляет GUID и связывает родительские `SyncId` через локальные JOIN.

### 6.3.2. Стратегия разрешения конфликтов

Применяется политика **«последняя запись побеждает»** (Last-Write-Wins) по полю `UpdatedAtUtc`. На сервере операция вставки-или-обновления выглядит как:

```sql
INSERT INTO Workouts (...) VALUES (...)
ON CONFLICT(SyncId) DO UPDATE SET
    Name = excluded.Name,
    UpdatedAtUtc = excluded.UpdatedAtUtc,
    ...
WHERE excluded.UpdatedAtUtc > Workouts.UpdatedAtUtc;
```

Условие `WHERE` гарантирует, что устаревший клиент не сможет затереть более свежие данные. Удаление реализовано как мягкое (поле `IsDeleted`) — это корректно распространяет факт удаления между устройствами.

### 6.3.3. Цикл синхронизации

Одна итерация (`SyncService.RunOnceAsync`) выполняется в четыре шага:

1. **Сбор push-батча.** Клиент выбирает из локальной БД все записи, у которых `UpdatedAt > LastSyncUtc` (время последней успешной синхронизации, хранимое в `Preferences`).
2. **Отправка.** `POST /sync` с JWT в заголовке `Authorization: Bearer`. Тело — три массива: `Workouts`, `WorkoutExercises`, `ExerciseSets`.
3. **Применение на сервере.** В рамках одной SQLite-транзакции сервер применяет батч и формирует ответный пакет, включая все свои записи, у которых `UpdatedAtUtc > LastSyncUtc`, плюс актуальное серверное время `ServerTimeUtc`.
4. **Применение ответа.** Клиент по `SyncId` определяет, существует ли запись локально, и либо вставляет новую, либо обновляет (если серверный `UpdatedAtUtc` свежее). Затем сохраняет `LastSyncUtc = ServerTimeUtc`.

Серверное время фиксируется до выборки исходящего батча — это исключает гонку, при которой запись, созданная сервером уже после её отбора, никогда не была бы загружена клиентом.

### 6.3.4. Безопасное хранение токена

JWT хранится на клиенте через `Microsoft.Maui.Storage.SecureStorage`. На Android используется `EncryptedSharedPreferences`, на iOS — Keychain, на Windows — DPAPI. Срок действия токена проверяется при каждом обращении к `GetTokenAsync`; за минуту до истечения токен удаляется и пользователю предлагается войти заново.

## 6.4. Развёртывание

### 6.4.1. Контейнеризация

Сервис упакован в Docker-образ с использованием двухстадийной сборки (multi-stage build). На стадии `build` применяется образ `mcr.microsoft.com/dotnet/sdk:9.0`, выполняется `dotnet restore` и `dotnet publish -c Release`. На стадии `runtime` используется минимальный `mcr.microsoft.com/dotnet/aspnet:9.0`, в который копируются только опубликованные артефакты. Такой подход уменьшает итоговый образ примерно в 4 раза и сокращает поверхность атаки.

Конфигурация задаётся через переменные окружения: `ASPNETCORE_URLS`, `DATABASE_URL`, `Jwt__Secret`, `Jwt__Issuer`, `Jwt__Audience`, `Jwt__ExpirationDays`. Это соответствует двенадцатифакторной методологии (12 Factor App) и упрощает миграцию между провайдерами хостинга.

### 6.4.2. База данных на сервере

В ходе разработки был выявлен архитектурный изъян первоначальной реализации: сервер использовал SQLite-файл в директории `/tmp`, которая на платформе Render является эфемерной — при каждом перезапуске контейнера (в том числе после выхода из режима сна) файл уничтожается. Следствием являлась потеря всех пользовательских данных при каждом «холодном старте», что делало межустройственную синхронизацию неработоспособной.

Для устранения проблемы хранилище данных на сервере заменено на **PostgreSQL**, предоставляемый платформой **Neon** (serverless PostgreSQL, бесплатный тариф — 512 МБ постоянного хранилища). Подключение реализовано через библиотеку `Npgsql`; строка подключения передаётся через переменную окружения `DATABASE_URL`, что полностью соответствует принципу хранения конфигурации вне кода (фактор III методологии 12 Factor App).

SQL-схема серверных таблиц (`Workouts`, `WorkoutExercises`, `ExerciseSets`, `Users`) полностью совместима с PostgreSQL без изменения бизнес-логики: конструкция `ON CONFLICT(SyncId) DO UPDATE SET ... WHERE excluded.UpdatedAtUtc > existing.UpdatedAtUtc` поддерживается PostgreSQL начиная с версии 9.5. Потребовалась лишь адаптация синтаксиса параметров запросов (`$name` → `@name`) и замена функции получения последнего вставленного идентификатора (`last_insert_rowid()` → `RETURNING Id`).

Данное решение устранило ключевое ограничение первоначального подхода: теперь данные сохраняются независимо от жизненного цикла контейнера, а синхронизация между несколькими устройствами работает корректно.

### 6.4.3. Хостинг

Сервис размещён на платформе Render.com по тарифу Free. Декларативная конфигурация описана в `render.yaml` (Blueprint). При коммите в основную ветку репозитория GitHub Render автоматически пересобирает образ и выкатывает обновление. Сертификат TLS выдаётся и обновляется автоматически.

Ограничения бесплатного тарифа отражены в реализации клиента: сервис засыпает после 15 минут неактивности, и первый запрос после сна занимает до 30 секунд. Чтобы пользователь не воспринимал паузу как зависание, в `AccountViewModel` реализован отложенный индикатор: если операция выполняется дольше 5 секунд, статус меняется на «Сервер засыпал, поднимаем (~30 сек)...». Таймаут `HttpClient` увеличен до 60 секунд.

## 6.5. Улучшение пользовательского опыта

Параллельно с развёртыванием выполнен пакет UX-улучшений, повышающих устойчивость приложения к ошибочному вводу и нестабильным сетевым условиям.

### 6.5.1. Дружественные сообщения об ошибках

Введён статический класс `NetworkErrors`, преобразующий низкоуровневые исключения в понятные пользователю сообщения на русском языке. Для исключений `HttpRequestException` и `TaskCanceledException` выдаются формулировки «Нет связи с сервером. Проверь интернет.» и «Сервер не отвечает. Попробуй ещё раз через минуту.» соответственно. Для ошибок аутентификации (`/auth/register`, `/auth/login`) тело ответа парсится как JSON, и пользователь получает контекстное сообщение — например, «Такой email уже зарегистрирован.» при HTTP 409.

### 6.5.2. Подтверждение деструктивных действий

Для операций, потенциально приводящих к потере данных, добавлены диалоги подтверждения через `Shell.Current.DisplayAlert`. В частности, при удалении упражнения из тренировки в диалоге явно указывается количество сетов, которые будут удалены: «Удалить упражнение и все его подходы (4)?». Такая формулировка снижает риск случайной потери записей при свайпах.

### 6.5.3. Валидация ввода в профиле

В разделе «Профиль» страницы настроек добавлены границы для веса тела (20–300 кг) и возраста (5–120 лет). Значения вне диапазона не сохраняются в базу, а поле подсвечивается красным. Это защищает не только базу данных от мусорных значений, но и ML-предсказатель `OnnxPredictionService`, использующий эти признаки.

### 6.5.4. Пустые состояния и pull-to-refresh

Для страниц со списками реализованы информативные пустые состояния (`CollectionView.EmptyView`) с иконкой и подсказкой о следующем действии. На странице шаблонов список обёрнут в `RefreshView` для обновления свайпом сверху вниз — стандартного жеста, ожидаемого пользователями мобильных приложений.

## 6.6. Результаты этапа

В рамках этапа реализованы и интегрированы:

- ASP.NET Core 9 API с тремя эндпоинтами, JWT-авторизацией и постоянным PostgreSQL-хранилищем (Neon);
- двунаправленная синхронизация с разрешением конфликтов по LWW и мягким удалением;
- безопасное хранение токена в `SecureStorage` платформы;
- автоматическое развёртывание в Render через Docker и GitHub-интеграцию;
- пакет UX-улучшений: обработка ошибок, индикаторы прогресса, валидация, подтверждения, пустые состояния.

В процессе тестирования была выявлена и устранена критическая проблема: эфемерная файловая система Render уничтожала SQLite-базу при каждом перезапуске контейнера. Переход на PostgreSQL через Neon обеспечил постоянное хранение данных и работоспособную синхронизацию между несколькими устройствами.

Приложение сохранило свойство offline-first: вся функциональность доступна без сервера, а синхронизация выступает дополнительной возможностью.
