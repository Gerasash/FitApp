# FitApp — ML-модуль

Эта папка содержит Python-код модуля прогнозирования: генератор синтетических
данных, эксперименты в Jupyter Notebook и обучение моделей. Результат обучения —
файл `.onnx`, который загружается в MAUI-приложение.

## Структура

```
ml/
├── generator/         # Параметрический симулятор виртуальных атлетов
│   ├── exercises.py   # Каталог упражнений с реалистичными нормативами
│   ├── athlete.py     # Класс VirtualAthlete: состояние и шаг симуляции
│   └── generate.py    # CLI: python -m ml.generator.generate ...
├── notebooks/         # Анализ и обучение (.py с маркерами ячеек # %%)
│   ├── 01_baselines.py  # Наивный и линейные предикторы — точка отсчёта
│   └── 02_lightgbm.py   # Признаки + обучение LightGBM, сравнение с baseline
├── export/            # Конвертация обученной модели в ONNX
│   └── to_onnx.py     # python -m ml.export.to_onnx (+ проверка эквивалентности)
├── data/
│   ├── synth.csv      # Сгенерированный датасет (не коммитится)
│   └── models/        # Обученная модель: .txt (LightGBM), .onnx, .meta.json
└── requirements.txt   # Python-зависимости
```

Готовая ONNX-модель (`lightgbm_1rm_28d.onnx`) и её метаданные
(`lightgbm_1rm_28d.meta.json`) копируются в `Resources/Raw/` MAUI-приложения,
откуда их загружает `Services/OnnxPredictionService.cs`.

## Первый запуск

Работаем в Python 3.11. Все команды — из корня репозитория FitApp.

```powershell
# 1. Создать виртуальное окружение
python -m venv ml/.venv

# 2. Активировать (Windows PowerShell)
ml\.venv\Scripts\Activate.ps1

# 3. Установить зависимости
pip install -r ml/requirements.txt

# 4. Сгенерировать пробный датасет (быстро, ~50 атлетов, 12 недель)
python -m ml.generator.generate --athletes 50 --weeks 12 --seed 42 --out ml/data/synth_small.csv
```

В VS Code: открой папку `FitApp` как корень, установи расширение **Python**,
выбери интерпретатор `ml/.venv/Scripts/python.exe`. Для notebook'ов также
поставь **Jupyter** extension.

## Параметры генератора

| Флаг | Значение по умолчанию | Описание |
|---|---|---|
| `--athletes` | 200 | Сколько виртуальных атлетов сгенерировать |
| `--weeks` | 26 | На сколько недель моделировать историю |
| `--seed` | 42 | Зерно ГПСЧ для воспроизводимости |
| `--out` | `ml/data/synth.csv` | Куда писать CSV |

Большой датасет (500 атлетов × 26 недель) генерируется за ~15 секунд, файл
~100 МБ, около 760 тыс. строк-подходов.

## Что генератор моделирует

Каждый виртуальный атлет — это набор параметров (пол, вес, возраст,
тренировочный стаж в месяцах, частота тренировок, индивидуальная скорость
прогресса, склонность к пропускам и к травмам). На старте по каждому
упражнению ему сэмплируется текущий 1ПМ из реалистичного распределения, исходя
из генетического потолка и пройденной доли пути к нему. Дальше неделя за
неделей моделируются тренировки: для каждого выбранного упражнения
рассчитывается рабочий вес на целевой RPE, генерируется 2–5 рабочих сетов
(+ разминка), считается фактический RPE с шумом. Между тренировками 1ПМ
постепенно растёт по логистической кривой — у новичков быстрее, у опытных
почти плато.

Помимо роста, симулятор моделирует и спады силы, без которых данные были бы
нереалистично «гладкими»:

- **Травмы.** В начале каждой недели разыгрывается вероятность травмы (с
  учётом индивидуальной склонности атлета). При срабатывании 1ПМ всех
  упражнений поражённой мышечной группы резко падает на 8–35 %, а сама группа
  выводится из тренировок на 1–3 недели восстановления. Крупные базовые группы
  (спина, ноги, грудь) травмируются чаще мелких.
- **Детренинг.** При длительной паузе по упражнению (более 10 дней) сила
  постепенно снижается — примерно на 1 % за неделю простоя, с поправкой на
  восстанавливаемость атлета.
- **Мышечная память.** Утраченную после травмы или паузы силу атлет
  возвращает заметно быстрее, чем набирал впервые: чем глубже просадка от
  ранее достигнутого пика, тем сильнее ускоряется восстановление.

Благодаря этому в данных присутствуют не только монотонные восходящие тренды,
но и просадки с последующим восстановлением — и модель учится прогнозировать
возврат к прежнему уровню, а не воспринимает падение как новую норму.

Подробности — в коде `ml/generator/athlete.py`.

## Работа с notebook'ами в VS Code

Notebook'и в этом проекте хранятся как `.py`-файлы с маркерами ячеек `# %%`.
Это удобнее для git-diff (обычный текст вместо JSON) и работает в VS Code:

1. Открой любой файл из `ml/notebooks/`.
2. Над каждой ячейкой будет ссылка **Run Cell** / **Run Below**, либо
   `Shift+Enter` выполнит текущую ячейку в Python Interactive Window.
3. При желании можно сконвертировать в классический `.ipynb`:
   ```
   pip install jupytext
   jupytext --to ipynb ml/notebooks/01_baselines.py
   ```

Все notebook'и можно запускать и просто как скрипты:
```
python ml/notebooks/01_baselines.py
```

## Задача и признаки

Модель прогнозирует **1ПМ (одноповторный максимум) данного упражнения данного
атлета через 28 дней**. Целевая переменная — `top_1rm` (оценка 1ПМ по формуле
Эпли) ближайшей тренировки, прошедшей не раньше чем через 28 дней от текущей.

Сырые подходы агрегируются в одну строку на (атлет, упражнение, дата), после
чего строятся 30 табличных признаков (все доступны на момент прогноза, утечки
из будущего нет):

- **История по упражнению:** лаги 1ПМ (`lag_1rm_1..5`), приращение `diff_1`,
  скользящие средние `mean_1rm_5` / `mean_rpe_5`, наклоны тренда `slope_3` /
  `slope_5`, исторический пик `max_1rm_hist` и просадка от него
  `drop_from_peak`, число прошлых тренировок, дни с первой и с прошлой
  тренировки.
- **Текущая тренировка:** `top_1rm`, рабочий вес, число сетов, статистики RPE
  и повторов.
- **Анкета атлета:** пол, вес, возраст, стаж, целевой RPE.
- **Описание упражнения:** идентификатор, базовое/изоляция, оборудование,
  мышечная группа (категориальные).

Признаки `max_1rm_hist` и `drop_from_peak` добавлены специально, чтобы модель
распознавала просадки после травм и пауз и прогнозировала восстановление к
ранее достигнутому уровню.

## Модель и метрики

Используется градиентный бустинг **LightGBM** (`objective=regression_l1`, MAE),
с ранней остановкой по отложенной выборке. Разбиение train/val/test — **по
атлетам** (70/10/20 %), чтобы исключить утечку истории одного человека между
выборками.

На тестовой выборке (500 атлетов × 26 недель):

| Модель | MAE, кг |
|---|---|
| Наивный (последнее значение) | ~2.4 |
| Линейная регрессия по 5 точкам | ~2.4 |
| **LightGBM** | **~1.8** |

Отдельно на сессиях с просадкой (текущий 1ПМ ≥ 10 % ниже исторического пика)
LightGBM даёт MAE ~1.3 кг против ~2.2 у наивного предиктора и корректно
прогнозирует восстановление вверх, а не застой на просевшем уровне.

## Инференс в приложении

Обученная модель конвертируется в **ONNX** (`python -m ml.export.to_onnx`) с
автоматической проверкой эквивалентности предсказаний LightGBM и ONNX. Файл
`.onnx` вместе с `.meta.json` копируется в `Resources/Raw/` и выполняется прямо
на устройстве через `Microsoft.ML.OnnxRuntime` — без сети и Python-сервиса.

Класс `Services/OnnxPredictionService.cs` строит вектор признаков из истории
тренировок пользователя **в том же порядке и по тем же формулам**, что и
`02_lightgbm.py` (порядок задан списком `feature_columns` в `.meta.json`).
Прогноз 1ПМ далее используется в `Services/WorkoutPlannerService.cs` для
рекомендации рабочего веса и повторов на следующую тренировку.

> При изменении набора или порядка признаков нужно синхронно править три
> места: `ml/notebooks/02_lightgbm.py`, блок проверки в `ml/export/to_onnx.py`
> и метод `BuildFeatures` в `OnnxPredictionService.cs`.

## Полный цикл переобучения

```powershell
# 1. Сгенерировать датасет
python -m ml.generator.generate --athletes 500 --weeks 26 --seed 42 --out ml/data/synth.csv

# 2. Обучить LightGBM (сохранит .txt и .meta.json в ml/data/models/)
python ml/notebooks/02_lightgbm.py

# 3. Сконвертировать в ONNX и проверить эквивалентность
python -m ml.export.to_onnx

# 4. Скопировать модель и метаданные в ресурсы приложения
copy ml\data\models\lightgbm_1rm_28d.onnx       Resources\Raw\
copy ml\data\models\lightgbm_1rm_28d.meta.json  Resources\Raw\
```

После обновления модели MAUI-приложение нужно пересобрать, чтобы новые файлы
попали в пакет.
